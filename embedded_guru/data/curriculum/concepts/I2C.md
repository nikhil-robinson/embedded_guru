# I2C — Inter-Integrated Circuit Protocol

## What it is

I2C is a two-wire synchronous serial protocol. Uses SDA (Serial Data) and SCL (Serial Clock). Multi-master, multi-slave. Address-based (7-bit or 10-bit addresses). Developed by Philips (now NXP). Governed by the I2C specification (UM10204).

## Electrical requirements

**Open-drain topology:** Both SDA and SCL are open-drain lines. Devices only pull lines LOW — they never drive HIGH. External pull-up resistors pull lines to VCC when no device is pulling them low.

**Pull-up resistor values:**
- Standard mode (100 kHz): 4.7 kΩ typical
- Fast mode (400 kHz): 2.2 kΩ typical
- Fast-mode Plus (1 MHz): 1 kΩ typical
- General safe range at 3.3V, 100kHz: 2.2 kΩ to 10 kΩ
- Minimum value is set by the maximum sink current of devices on the bus (typically 3 mA) and VCC: R_min = VCC / I_sink_max

**Common mistake — 1-ohm pullup:** Using 1 Ω or very low resistors to "make the signal stronger" is WRONG. With 1 Ω at 3.3V: current = 3.3V / 1 Ω = 3.3 A. This far exceeds the GPIO sink current limit (25 mA on STM32F4). Risk: bus contention, GPIO damage, line never reaching valid HIGH logic level. The 4.7 kΩ value is not convention — it is the result of I2C spec timing constraints on rise time (C_bus × R_pullup ≤ t_rise_max).

## Protocol operation

**Bus idle:** Both SDA and SCL HIGH.

**START condition:** SDA falls while SCL is HIGH. Generated by master.

**Address frame:** 7-bit device address + R/W bit. Master transmits, all slaves compare to their address. Matching slave responds with ACK (pulls SDA low during 9th clock).

**Data frames:** 8 bits of data + ACK bit. Direction set by R/W bit in address frame.

**STOP condition:** SDA rises while SCL is HIGH. Returns bus to idle.

**NACK:** No device pulled SDA low during ACK clock pulse. Reasons: wrong address, device not ready, master signaling end of read transaction (intentional NACK on last byte from master).

**Repeated START:** Master issues START without issuing STOP first. Used to change direction (write address register, repeated-start, read data) without releasing the bus.

## Addressing

7-bit addressing: 128 possible addresses, 16 reserved by spec. Common sensor addresses:
- LM75 temperature sensor: 0x48–0x4F (3 address pins)
- TMP102: 0x48–0x4B
- SHT31: 0x44 or 0x45
- BME280: 0x76 or 0x77
- MPU6050 IMU: 0x68 or 0x69

Write address byte = (device_address << 1) | 0 (R/W=0)
Read address byte = (device_address << 1) | 1 (R/W=1)

Example: LM75 at address 0x48 → write = 0x90, read = 0x91

## Register read sequence (I2C master read, typical sensor)

1. START
2. Send address byte with R/W=0 (write)
3. Wait for ACK
4. Send register pointer byte (e.g. 0x00 for temperature register)
5. Wait for ACK
6. Repeated START
7. Send address byte with R/W=1 (read)
8. Wait for ACK
9. Clock in first data byte, send ACK (if more bytes follow)
10. Clock in second data byte, send NACK (signals end of read to slave)
11. STOP

**Common mistake — forgetting pointer register:** Reading without sending the register pointer first will read whatever register the internal pointer was last set to — unpredictable. Always write the register address before reading.

## STM32 HAL vs raw register access

**HAL_I2C_Mem_Read:** Single function hides all of the above — sets address, sends register pointer, restarts, reads bytes. Appropriate for production code where clarity matters. Inappropriate for learning — student does not understand what bytes travel on the wire.

**Acceptable primitives for Milestone 0 (L2):** HAL_I2C_Master_Transmit / HAL_I2C_Master_Receive. Student must sequence the calls correctly and understand each does START → address → data → STOP.

**Bare register (L3 Milestone 0):** Drive I2C_CR1, I2C_CR2, I2C_SR1, I2C_SR2 directly. Steps: enable I2C clock (RCC_APB1ENR bit), configure GPIO AF (open-drain, pull-up, speed), set I2C_CR2 FREQ bits, set CCR and TRISE registers, enable with PE bit in I2C_CR1, then generate START/STOP manually via SB/BTF/ADDR bits.

## Key STM32F4 registers (I2C1, base 0x40005400)

| Register | Offset | Key fields |
|---|---|---|
| I2C_CR1 | 0x00 | PE (enable), START, STOP, ACK, SWRST |
| I2C_CR2 | 0x04 | FREQ[5:0] — APB1 clock in MHz |
| I2C_OAR1 | 0x08 | Device's own address (slave mode) |
| I2C_DR | 0x10 | Data register — read/write data byte |
| I2C_SR1 | 0x14 | SB (start sent), ADDR (address ack'd), TXE (TX empty), RXNE (RX not empty), BTF, BERR, ARLO |
| I2C_SR2 | 0x18 | BUSY, TRA (transmitter), MSL (master mode) |
| I2C_CCR | 0x1C | Clock control — set for standard/fast mode |
| I2C_TRISE | 0x20 | Maximum rise time |

## Clock enable (STM32F4, required before any I2C config)

I2C1: Set bit 21 of RCC_APB1ENR (address 0x40023840)
I2C2: Set bit 22 of RCC_APB1ENR
I2C3: Set bit 23 of RCC_APB1ENR

**Common mistake:** Configuring I2C registers before enabling the peripheral clock via RCC. Result: register writes silently ignored, peripheral never initializes. Reading back the register before and after RCC enable will show the value stays 0 until clock is enabled.

## Common debugging checks when I2C doesn't work

1. Is the I2C peripheral clock enabled in RCC?
2. Is the GPIO configured as alternate function, open-drain, with internal or external pull-up?
3. Is the GPIO AF number correct? (AF4 for I2C1/2/3 on STM32F4 — verify in datasheet, not assumed)
4. Did you write the register pointer byte before the read?
5. Is the pull-up resistor value in range (2.2k–10k at 3.3V/100kHz)?
6. Is the device address correct? (Most common mistake — verify against sensor datasheet)
7. Are SCL and SDA pins not swapped?
8. Is the I2C bus stuck? (BUSY flag set, won't clear) — master must generate 9 clock pulses to unstick, or assert SWRST in I2C_CR1.
